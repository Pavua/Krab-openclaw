# Второй Deep Research: Что Применимо Для Krab / OpenClaw

Источник: `/Users/USER3/Downloads/deep-research-report.md`

## Зачем этот документ в handoff

Этот отчёт полезен не тем, что предлагает «волшебно подешевле модели», а тем, что правильно ставит акцент на архитектуре расходов: у OpenClaw деньги сгорают не на одном большом ответе, а на множестве фоновых LLM-циклов, инструментах и повторной прокачке контекста.

## Главный вывод отчёта

Для Krab / OpenClaw оптимизация должна идти по четырём направлениям:
- уменьшать число дорогих LLM-циклов;
- уменьшать размер каждого цикла;
- переводить детерминируемые и потоковые участки в локальные / не-LLM пайплайны;
- заменять heartbeat-петли на event-driven поведение там, где это реально возможно.

## Что из отчёта стоит принять практически

### 1. Cost explosion идёт от runtime-петель, а не от «длины ответа»

Это особенно важно для нашей архитектуры, потому что у нас уже есть:
- owner panel;
- userbot;
- translator / voice loops;
- multi-channel и planned swarm / bus контуры.

Вывод для проекта:
- нужно экономить не только на модели, а на числе вызовов и размере контекста;
- diagnostics, ACL, capability truth, health и panel refresh должны по максимуму быть детерминированными или дешёвыми.

### 2. Event-driven лучше heartbeat-петли

Отчёт правильно подчёркивает, что фоновая «проверка ради проверки» быстро превращается в токенный пожар.

Применение к Krab:
- inbox / escalation / transport hooks стоит усиливать как event-driven слой;
- housekeeping должен оставаться только там, где он реально повышает надёжность;
- silent loops без пользы нужно постепенно вычищать.

### 3. Compaction / pruning / memory flush — это не косметика, а FinOps-ядро

Из отчёта полезны именно эти принципы:
- pruning tool results;
- разумный compaction;
- memory flush перед свёрткой;
- явный контроль context growth.

Для Krab это значит:
- tool outputs не должны бесконтрольно раздувать active context;
- надо внимательнее смотреть на всё, что попадает в видимый runtime-context в OpenClaw;
- owner panel должна со временем показывать хотя бы часть truthful context/cost drift сигналов.

### 4. Codex OAuth — да, ChatGPT Web «как API» — нет

Отчёт здесь хорошо совпадает с нашей линией:
- легитимный путь использования оплаченной подписки — `openai-codex` как поддержанный провайдерный контур;
- не стоит превращать consumer web UI в основную production-схему для агента.

Применение к нашему проекту:
- `openai-codex` остаётся нормальным supported-вариантом;
- web/session-backed consumer мосты — только как lab / fallback и только с честной маркировкой риска.

### 5. Google Antigravity / похожие consumer-интеграции — только как lab

Это особенно ценно для нашего runtime design:
- не использовать как primary;
- не обещать надёжность там, где её не может быть;
- держать в отдельном experimental контуре.

### 6. На M4 Max 36GB нельзя строить стратегию вокруг 70B baseline

Это очень прикладной вывод.

Для нашего железа и наших задач реалистичнее:
- 13B/14B как cheap local tier;
- около 32B как upper local working tier;
- не пытаться строить стабильный 24/7 план вокруг 70B на 36GB unified memory.

Это полезно для USER3 и для возврата на `pablito`, потому что стратегия должна быть переносимой, а не держаться на случайном «вроде запускается».

### 7. Для voice/translator LLM должен быть semantic post-processor, а не весь конвейер

Это один из самых практичных выводов отчёта.

Применение к нашему translator-треку:
- STT / language detect / timeline / буферы / часть diarization лучше держать локально и детерминированно;
- LLM использовать точечно:
  - перевод сложных сегментов,
  - терминология,
  - нормализация,
  - summaries.

Это прямо соответствует нашему желанию экономить без потери качества.

### 8. Нужен cost dashboard по runtime-сущностям, а не только общая сумма

Отчёт предлагает здравую гранулярность:
- `agentId`
- `sessionKey`
- `team_id`
- `channel_id`

Для Krab это очень подходит.

Минимальный полезный следующий уровень observability:
- route-class;
- provider-class;
- contextTokens;
- частота compaction;
- доля tool-heavy циклов;
- доля silent housekeeping.

## Что из отчёта не стоит принимать буквально

### 1. Не надо превращать экономию в культ «только local»

Отчёт полезен, но если довести идею до крайности, можно испортить качество.

Правильный вывод для нас:
- local — это operator / cheap tier;
- cloud — это deep reasoning / high-ambiguity / quality-critical слой.

### 2. Не надо строить продуктовую стратегию на “серых” мостах и reverse-proxy

Даже если технически это временами работает, для нашего проекта это плохой production foundation.

### 3. Не стоит путать “стриминговый UX” с настоящим provider streaming

Это мы уже начали чинить честно. Нужно продолжать в том же духе.

## Что это меняет в нашем ближайшем плане

### Обязательно ввести позже
- route classes в model routing;
- complexity gate перед cloud escalation;
- truthful cost / route observability в owner panel;
- pruning и compaction как FinOps-инструмент, а не как второстепенную функцию.

### Обязательно не ломать
- multi-account модель: runtime per-account, repo shared;
- совместимость USER3 -> pablito;
- сохранение OAuth/auth-вариантов;
- truthful UI вместо “красивых, но ложных” статусов.

## Короткий вывод

Этот второй отчёт усиливает правильную мысль: экономия для OpenClaw приходит не из дешёвого хака, а из грамотной архитектуры циклов, контекста и маршрутизации.

Для Krab / OpenClaw это значит:
- меньше дорогих LLM-петель;
- больше deterministic и local-first там, где это разумно;
- cloud только там, где он реально приносит качество;
- честная observability по cost и routing прямо в owner panel.
