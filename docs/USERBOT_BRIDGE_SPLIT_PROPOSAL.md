> **Origin:** Generated by a `feature-dev:code-architect` subagent during session 4
> (2026-04-09) as part of a parallel agent experiment. This is a **proposal**, not an
> approved plan — requires review and approval before implementation in session 5+.
>
> **Agent metrics:** 21k tokens, 11 tool uses, 97s wall-clock, single round.
> The agent got stuck in an initial read loop due to read-only tool limitations
> (`feature-dev:code-architect` has no Bash/Write/Edit) but produced this full

## Problem Statement

`src/userbot_bridge.py` is 6000+ lines in a single file, mixing:
- Pyrogram session lifecycle and client management
- LLM request pipeline with streaming, timeouts, progress notices
- Background task lifecycle and reaper
- Text processing (reasoning strip, batching, splitting)
- Voice runtime profile and delivery
- Runtime status fast paths (capability questions, model questions)
- ACL and access control
- Misc helpers

This mixing creates:
- Review fatigue (can't audit a 6000-line file systematically)
- Test fragility (touching one area risks breaking another)
- Merge conflict magnets in any multi-agent workflow
- Hard to find pre-existing style warnings (`W293`, `N806`)

## Proposed Architecture

**Strategy:** Hybrid (by-feature + infrastructure-layer separation) using **mixin-based
extraction**. Each module defines a mixin class (e.g. `LLMFlowMixin`, `VoiceProfileMixin`)
that `KraabUserbot` inherits from, preserving the `self.*` namespace and avoiding
circular imports.

## Migration Strategy

### Ordering (safest first)

1. **`llm_text_processing.py`** — stateless, well-tested, can be moved in
   an afternoon with zero production risk.
2. **`runtime_status.py`** — config-only reads, no shared state.
3. **`voice_profile.py`** — already has singleton-extraction pattern from
   session 4 (`chat_ban_cache`, `chat_capability_cache`).
4. **`access_control.py`** — pure functions, easy to isolate.
5. **`llm_flow.py`** — medium risk; move after `background_tasks.py` is
   extracted so its dependencies are clearer.
6. **`background_tasks.py`** — careful work on shared mutable state.
7. **`session.py`** — last; touching Pyrogram lifecycle during an active
   production migration is risky.

### Mixin extraction pattern

To avoid circular imports and preserve `self.*` access, each module provides
a mixin class:

```python
# src/userbot/llm_flow.py
class LLMFlowMixin:
    async def _run_llm_request_flow(self, ...):
        # same body as before, references self.*
        ...
    
    async def _finish_ai_request_background(self, **kwargs):
        ...
```

And `userbot_bridge.py`:

```python
from src.userbot.llm_flow import LLMFlowMixin
from src.userbot.background_tasks import BackgroundTasksMixin
from src.userbot.session import SessionMixin
# ... etc

class KraabUserbot(
    LLMFlowMixin,
    BackgroundTasksMixin,
    SessionMixin,
    VoiceProfileMixin,
    AccessControlMixin,
    RuntimeStatusMixin,
    LLMTextProcessingMixin,
):
    """Main orchestrator. Most methods now inherited from mixins."""
    def __init__(self): ...
    async def start(self): ...
    async def _process_message(self, message): ...
    # ...
```

This preserves:
- `self.*` single namespace
- No circular imports (mixins don't import each other)
- Existing `KraabUserbot.__new__()` test pattern continues to work
- Backward compat: `from src.userbot_bridge import KraabUserbot` unchanged

## Risks and Mitigations

### RISK 1: Circular dependency `llm_flow.py` ↔ `background_tasks.py`

`_run_llm_request_flow` calls `_register_chat_background_task` and
`_get_active_chat_background_task`. `_background_task_reaper` calls
`_finish_ai_request_background`.

**Mitigation:** Mixins reference `self.*` — no import between modules needed.

### RISK 2: Shared mutable state breakage

`_chat_background_tasks` dict is mutated from at least 3 places.

**Mitigation:** Move the dict initialization to `KraabUserbot.__init__`. Mixin
methods reference `self._chat_background_tasks` — dict lives in the instance,
mixins just operate on it.

### RISK 3: `_run_llm_request_flow` is a 757-line method

That's a single method bigger than many modules in this project.

**Mitigation:** After extraction to `llm_flow.py`, a follow-up refactor can
split it internally into smaller helpers. But the initial split should move
it verbatim to avoid bundling refactoring with extraction (reduces review
surface).

### RISK 4: Test impact

~15 `test_userbot_*.py` files import `KraabUserbot` via
`from src.userbot_bridge import KraabUserbot`.

**Mitigation:** With mixin pattern, `KraabUserbot` class still lives in
`userbot_bridge.py` — the import doesn't change. Only ~3-4 tests that import
module-level helpers (like `_RELAY_INTENT_KEYWORDS`, `_build_openclaw_*`)
need updating.

## Validation Plan

Per migration step:

1. Move methods to new mixin file
2. Add `KraabUserbot(NewMixin, ...)` in `userbot_bridge.py`
3. Delete moved methods from `userbot_bridge.py`
4. Run full sweep: `venv/bin/python -m pytest tests/unit/ -q`
5. Must remain green before moving to next module
6. Smoke test via `mcp__krab-yung-nagato__krab_status` after restart
7. Commit per module (7 separate PRs or 7 commits in one feature branch)
