# Krab Branch Handoff Governor

Используй этот навык, когда важно не просто написать код, а сохранить понятную историю владения: кто что менял, в какой ветке, в каком порядке мерджить и какие проверки относятся к каждому куску работы.

Этот навык особенно полезен, когда одновременно живут `pablito`, `USER2`, `USER3`, несколько Codex/Claude-сессий и незакоммиченные локальные изменения.

## Базовая модель веток

- Для каждой новой смысловой задачи предпочитай отдельную ветку `codex/...`.
- Не используй одну и ту же ветку для несвязанных экспериментов.
- Если несколько агентов работают параллельно, каждому нужен либо свой branch, либо жёстко разделённый write-scope.
- Dirty worktree перед стартом надо классифицировать:
  - `моё по текущей задаче`
  - `чужое или ранее существовавшее`
  - `generated / artifacts`

## Рабочий цикл

1. Зафиксируй стартовую точку:
   - текущая ветка;
   - есть ли локальные изменения;
   - кто владелец следующего acceptance verdict.
2. Определи lane ownership:
   - какие файлы может менять этот агент;
   - какие файлы зарезервированы за другими.
3. Перед handoff оставь краткую truthful note:
   - что изменено;
   - что проверено;
   - что ещё рискованно или не проверено.
4. Перед merge собери merge-order:
   - infra/shared contracts;
   - feature implementation;
   - docs/artifacts;
   - финальные acceptance fixes.
5. Если есть сомнение по drift или ownership, останови merge и сначала синхронизируй truth.

## Что должно быть в handoff note

- branch name;
- текущая учётка;
- список изменённых областей;
- что реально проверено;
- что не трогалось специально;
- какие артефакты или тесты считать source-of-truth.

## Merge guardrails

- Не мерджи ветку, пока не понял, какие незакоммиченные изменения были до тебя.
- Не выдавай helper-account smoke за финальный release verdict.
- Не смешивай в одном handoff и код, и runtime-state, если runtime не принадлежал этой сессии.
- Не переписывай чужие изменения только ради чистоты истории.

## Красные флаги

- Один и тот же файл редактируют два агента без явного ownership.
- Ветка используется и для feature, и для аварийного hotfix, и для docs cleanup сразу.
- Handoff говорит "всё готово", но не содержит branch, verification и остаточные риски.
- Передача между `USER2` / `USER3` / `pablito` идёт без фиксации текущего владельца runtime.

## Рекомендуемые связки с другими skills

- `krab-parallel-agent-orchestrator` для разрезания по lane до начала работ.
- `krab-role-split-delivery` для крупных межслойных задач.
