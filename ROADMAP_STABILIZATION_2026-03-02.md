# План стабилизации и развития Krab/OpenClaw (срез на 2 марта 2026)

## Цель
Сделать экосистему предсказуемой и отказоустойчивой: один клик запуска, стабильные каналы OpenClaw, корректный fallback cloud/local, единая диагностика в web-панели.

## Текущее состояние
- `cloud-chain`: ключи Gemini рабочие, базовая диагностика проходит.
- `userbot`: отвечает через локальную модель, но остаются редкие проблемы с Telegram-сессией и перезапусками.
- `telegram bot/iMessage/прочие каналы OpenClaw`: есть деградации при выгрузке локальной модели и в отдельных сессиях.
- `voice_gateway` и `krab_ear`: интегрированы в runtime-код как реальные клиенты health, но сами сервисы нужно держать поднятыми.

## Этап 1 — Стабилизация каналов OpenClaw (высший приоритет)
1. Зафиксировать единые DM-policy правила для всех каналов (`telegram/imessage/whatsapp/signal/discord/slack`) через `openclaw_runtime_repair`.
2. Довести до нуля pairing-спам и «no models loaded» в direct-диалогах.
3. Вычистить залипшие session overrides при старте и по кнопке из web-панели.
4. Добавить регрессионный smoke по каналам (минимум Telegram bot + iMessage + WhatsApp).

Критерий выхода:
- Каналы не отправляют pairing-спам без запроса.
- Ответ приходит стабильно даже после auto-unload локальной модели.

## Этап 2 — Надёжность Telegram userbot-сессии
1. Убрать агрессивный purge session по timeout (сделано в коде).
2. Финализировать мягкий stop/start в `.command` скриптах (частично сделано).
3. Добавить технический watchdog-тест на 5-10 перезапусков без ручного логина.
4. Зафиксировать recovery-политику: purge только при `auth_key_unregistered/auth key not found`.

Критерий выхода:
- Перезапуск через `new Stop Krab.command` -> `new start_krab.command` не требует повторного логина в большинстве циклов.

## Этап 3 — Полная интеграция соседних проектов
1. Поднять `Krab Voice Gateway` и `Krab Ear` как обязательные сервисы экосистемы.
2. Добавить one-click orchestrator для всего стека (Krab + OpenClaw + Voice Gateway + Krab Ear).
3. Подключить расширенную диагностику по voice/ear в web-панели.
4. Уточнить контракт между проектами: URL, API key, health, retry, timeout.

Критерий выхода:
- `/api/ecosystem/health` отражает реальные состояния всех 4 компонентов, без `not_configured`.

## Этап 4 — Маршрутизация и UX модели
1. Закрепить `nvidia/nemotron-3-nano` как default local model.
2. Ввести явный профиль vision-модели отдельно от текстовой (без лишней RAM-нагрузки).
3. Доделать `!model scan` с размером модели и нормальным выбором конкретной модели.
4. Исправить ответ «какая модель фактически использована» по runtime-route, а не по self-report LLM.

Критерий выхода:
- Команда статуса модели всегда показывает фактический runtime канал/модель.

## Этап 5 — OpenClaw Dashboard/Browser relay/skills
1. Добить стабильность Chrome Relay (порт, attach state, diagnostics endpoint).
2. Разобрать blocked skills по фактическим причинам (bin/env/auth) и дать отдельный runbook.
3. Убрать дублирование функционала между Krab web-panel и встроенным OpenClaw UI.

Критерий выхода:
- Базовые кнопки dashboard + relay работают без ручных «танцев» после перезапуска.

## Этап 6 — Релизный цикл и git-процесс
1. После каждого завершённого этапа: commit -> push -> smoke -> PR -> merge в `main`.
2. Новый этап вести в отдельной ветке `codex/...`.
3. Поддерживать журнал изменений и rollback-команды в `PROJECT_STATE_MIGRATION.md`.

Критерий выхода:
- Есть стабильные точки отката и воспроизводимый one-click запуск.

## Когда менять чат и включать Plan mode
- Меняем чат, когда:
  1) закрыт текущий подэтап (код + тесты + commit/push),
  2) нужно переключаться на крупный архитектурный блок (интеграция нескольких репозиториев),
  3) контекст диалога стал слишком шумным для точной работы.
- Включаем `Plan mode`, когда:
  1) нужно согласовать multi-step миграцию между `Krab` / `Krab Voice Gateway` / `Krab Ear`,
  2) есть риск затронуть несколько контуров сразу (runtime, web, channels, infra),
  3) нужен формальный checkpoint с критериями приемки перед началом изменений.
